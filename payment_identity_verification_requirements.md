# 支付前身份验证 — 需求

> 编写日期：2026-05-14
> 来源：PRD V0.13（`PRD.docx` 1.4.5 + 1.4.1）+ 用户 2026-05-14 完整流程图
> 适用对象：PM / 业务方 / 开发评审
> 配套实施方案：[`payment_identity_verification_plan.md`](payment_identity_verification_plan.md)

## 1. 背景

现状：顾客扫描员工生成的支付二维码进入 `payment_entry` 落地页后，**直接调起微信支付**，整个流程没有任何会员匹配、手机号验证、代付识别。

问题：
- 违反 PRD「会员匹配仅在支付时进行 + 代付订单必须验证手机号授权」
- 老订单 `order.member_id` 不准：开单时录入的姓名/手机号未与会员体系打通
- 储值/券/积分等会员权益无法在支付时生效

目标：在顾客扫码进入支付页后、调起付款前，根据"开单时是否匹配会员 × 扫码方是否为会员 × 支付通道（微信/支付宝）"三个维度做账户匹配，必要时弹出二次确认或一键手机号授权。

## 2. 顶层不变项

1. **开单环节不做会员匹配** —— 员工录入顾客姓名/手机号/性别即可成单，不强制找会员
2. **会员匹配 / 储值 / 券使用都在「支付时」发生** —— 由扫码方身份决定
3. **代付订单必须做手机号授权验证** —— 防止恶意挂账
4. **支付宝支付订单一律标记「微信未验证」** —— 业务上还是希望最终能拿到顾客微信账号

## 3. 概念定义

| 概念 | 含义 |
|---|---|
| 开单方 | 员工，录入订单的人 |
| 扫码方 / 付款方 | 顾客，扫支付二维码进入 `payment_entry` 的人 |
| 订单归属（`order.member_id`） | 会员中心看到这一单时显示为"我"的会员 |
| 正常支付 | 订单归属切换到付款方。常见：未匹配会员 + 已是会员 ⇒ 直接归扫码方 |
| 替人代付 | 订单归属**保留**为开单时匹配的原会员，仅记录这笔付款是 xx 代付的 |
| 微信未验证 | 订单上的标志：该单尚未与微信账户（openid/unionid）建立关联。支付宝支付的订单一律设此标志；后续顾客用微信小程序登录该手机号会员后可清除 |
| 第三方 ID | 微信支付：openid + unionid；支付宝支付：payerid |

## 4. 完整决策流程

按"订单是否已匹配会员"分两大主分支，再按支付通道分两小分支，每条最终归宿都明确写明订单归属和「微信未验证」标志。

### 4.1 当前订单**未匹配**到会员

#### 4.1.1 微信支付
1. 通过 openid / unionid 查会员
   - **是会员** ⇒ 支付成功后，订单归扫码方
   - **非会员** ⇒ 用微信小程序 `getPhoneNumber` 拿手机号再查
     - **找到** ⇒ 支付成功后，订单归扫码方
     - **未找到** ⇒ 注册新会员，订单归该新会员

#### 4.1.2 支付宝支付（**全部**额外标"微信未验证"）
1. 通过 payerid 查会员
   - **是会员** ⇒ 支付成功后，订单归扫码方，并标「微信未验证」
   - **非会员** ⇒ 用支付宝小程序拿手机号再查
     - **找到** ⇒ 支付成功后，订单归扫码方，并标「微信未验证」
     - **未找到** ⇒ 注册新会员（来源为支付宝），订单归该新会员，并标「微信未验证」

### 4.2 当前订单**已匹配**到会员

#### 4.2.1 微信支付
1. 通过 openid / unionid 查扫码方是否为会员
   - **是会员**
     - 与订单匹配会员**一致** ⇒ 正常支付
     - **不一致** ⇒ **询问 二选一**：
       - 选「替人代付」 ⇒ 订单归属保留为原会员
       - 选「正常支付」 ⇒ 订单归属切到扫码方会员
   - **非会员**
     - 用微信小程序拿手机号查
       - **找到**
         - 与订单匹配会员**一致** ⇒ 正常支付
         - **不一致** ⇒ **询问 二选一**：
           - 选「替人代付」 ⇒ 订单归原会员
           - 选「正常支付」 ⇒ 订单归扫码方会员
       - **未找到** ⇒ **询问 二选一**：
         - 选「替人代付」 ⇒ 订单归原会员
         - 选「正常支付」 ⇒ 注册新会员，订单归新会员

#### 4.2.2 支付宝支付（**全部**额外标"微信未验证"）
1. 通过 payerid 查扫码方是否为会员
   - **是会员**
     - 与订单匹配会员**一致** ⇒ 正常支付 + 标「微信未验证」
     - **不一致** ⇒ **询问 二选一**：
       - 选「替人代付」 ⇒ 订单归原会员 + 标「微信未验证」
       - 选「正常支付」 ⇒ 订单归扫码方会员 + 标「微信未验证」
   - **非会员**
     - 用支付宝小程序拿手机号查
       - **找到**
         - 与订单匹配会员**一致** ⇒ 正常支付 + 标「微信未验证」
         - **不一致** ⇒ **询问 二选一**：
           - 选「替人代付」 ⇒ 订单归原会员 + 标「微信未验证」
           - 选「正常支付」 ⇒ 订单归扫码方会员 + 标「微信未验证」
       - **未找到** ⇒ **询问 二选一**：
         - 选「替人代付」 ⇒ 订单归原会员 + 标「微信未验证」
         - 选「正常支付」 ⇒ 用支付宝注册新会员，订单归新会员 + 标「微信未验证」

## 5. 状态机摘要

把第 4 节决策结果归并为 5 种 status，供前端在支付页驱动 UI：

| status | UI 表现 | 用户动作 | 后续 |
|---|---|---|---|
| `error` | 红字错误提示，无支付按钮 | — | — |
| `phone_required` | "请先验证手机号"+ 一键授权按钮 | 微信走 `getPhoneNumber` / 支付宝走 `my.getAuthCode` | 拿到手机号后重判，转入其他 status |
| `direct` | 默认"敬请支付"按钮 | 点击 → 调起付款 | 直接调原有支付流程 |
| `direct_to_scanner` | "订单将归您 138****1234"+ 确认按钮 | 点确认 → 锚定归属 | 转 `direct` |
| `choose_identity` | 弹窗二选一：①归我（正常支付）②替人代付 + 取消 | 选①或② | 写入对应归属 + 代付标志后转 `direct` |

支付宝支付时所有 status 的后续动作都额外写入「微信未验证」=true。

## 6. 数据字段需求

只描述需求层面，具体表 / 类型 / 索引设计见实施方案。

| 字段 | 归属 | 用途 | 备注 |
|---|---|---|---|
| 订单归属会员 ID | 订单（已存在） | 会员中心列表 / 储值 / 券 / 积分使用 | 「正常支付」时切换；「替人代付」时保留 |
| 微信未验证标志 | 订单（新增） | 标记订单尚未与微信账户关联，供后续运营触达"建议绑定微信" | 支付宝支付一律设；后续微信验证完成后清除 |
| 付款方会员 ID | 付款记录（已存在） | 记录这笔付款实际由谁付 | 「正常支付」= 订单归属 = 付款方；「替人代付」时不同 |
| 代付标志 | 付款记录（新增） | 标记该笔付款是代付 | `订单归属会员 ≠ 付款方会员` 时为 true |

## 7. 关键边界 / 例外

- **手机号已被另一会员认证（PRD 1.4.1 冲突规则）**：
  - 当前用微信 ID 试图绑：若该会员已绑微信 ⇒ 拒绝（提示「该手机号已绑其他账户，请换号或换微信」）；未绑微信 ⇒ 把当前微信 ID 链到该会员
  - 当前用支付宝 ID 试图绑：若该会员已绑支付宝 ⇒ 拒绝；未绑支付宝 ⇒ 把 payerid 链到该会员
- **代付订单的退款**：按 PRD 1.4.5「代付订单退款原路退回至支付账户」，退到付款方账户，不退到订单归属会员账户
- **微信开发者工具 `getPhoneNumber` 不返真实号**：测试需真机；开发期间建议保留 `?mockCell=` 后门
- **支付宝小程序对接未完成**：本期支付宝手机号解密通道未通；后端按 stub 处理（接受 mock 手机号字段或返"待对接"错误码）
- **幂等**：扫码页可能被刷新 / 双击。决策接口要支持反复查询不变；落库接口要在"已锚定身份"时直接返既有结果，不重写

## 8. 验收清单（业务视角）

1. 老订单（开单未填手机号）顾客扫微信支付二维码 ⇒ 直接归扫码方（4.1.1 路径），订单上有「订单归属」字段
2. 老订单（开单填了 138****1234 但无会员）顾客扫微信支付 ⇒ 同上（视为未匹配，归扫码方）
3. 顾客 A 开单时录入手机号匹配会员 A，顾客 B 用自己微信扫码 ⇒ 弹"替人代付 / 正常支付"二选一（4.2.1 路径）
4. 同上场景但 B 选支付宝支付 ⇒ 仍弹二选一，且不论选哪种最后订单都有「微信未验证」标志
5. 新顾客扫码 + 微信未授权过手机号 ⇒ 先一键授权 ⇒ 进入对应路径
6. 已确认过身份的订单，顾客刷新支付页 ⇒ 不再弹身份选择，直接到「敬请支付」
7. 代付的支付完成后退款 ⇒ 退到付款方账户（不在本期实施范围，验收时仅文档化）

## 9. 不在本期实施范围

- 支付宝真实手机号解密（接 `alipay.system.oauth.token` + `alipay.user.info.share`）
- 「微信未验证」订单后续"完成验证"的入口（小程序登录该会员后清除标志的页面）
- 退款分流（代付订单退款原路退回付款方账户）
- 决策时机改为"支付成功后才真正落地"语义（本期为简化采用"用户确认即落地"；如业务实测发现"用户中途放弃"导致归属错误较多再改）
